01

Core Concepts

Docker packages applications into containers — lightweight, isolated environments that share the host OS kernel. Unlike VMs, containers start in milliseconds and use minimal resources. The Docker daemon manages the container lifecycle: build, run, stop, remove.

Images

Read-only templates built from Dockerfiles. Layered filesystem — each instruction creates a cacheable layer. Base images come from registries like Docker Hub.

Containers

Running instances of images. Ephemeral by default — data disappears when container stops unless persisted to volumes or bind mounts.

Volumes

Persistent storage managed by Docker. Survives container restarts. Named volumes for databases, bind mounts for development hot-reload.

Compose

Multi-container orchestration via docker-compose.yml. Define services, networks, and volumes in one file. docker compose up starts everything.

02

Images & Multi-Stage Builds

Multi-stage builds use multiple FROM statements to separate build dependencies from runtime. Build stage compiles code with full toolchain, final stage copies only the binary into a minimal base image. Result: images 10-100x smaller.

Dockerfile (multi-stage)
# Build stage
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Runtime stage
FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
EXPOSE 80
Tip: Order Dockerfile instructions from least to most frequently changing. Package install before source copy — Docker caches unchanged layers.
03

Networking

Docker creates isolated networks. Containers on the same network communicate by service name. Bridge network for single-host, overlay for Swarm/multi-host. Port mapping (-p 8080:80) exposes container ports to the host.

Bridge

Default network. Containers get internal IPs, communicate via container name on same bridge. Good for local development.

Host

Container shares host network stack directly. No port mapping needed. Maximum performance, no isolation.
